# Football Predictions — Световно първенство 2026

Уеб приложение за управление и следене на футболни прогнози за Световното първенство по футбол 2026. Всеки участник въвежда своите прогнози за крайните резултати на мачовете, а системата автоматично изчислява точките и поддържа актуално класиране в реално време. Изградено с **Angular 21** за фронтенда и **Supabase (PostgreSQL)** за базата данни, с поддръжка на **Socket.io** за live обновяване на резултатите от външен API.

## Функционалност

### За всички потребители

- **Главна таблица с прогнози** — показва всички прогнози на всички участници, наредени по мачове и фази. Всяка клетка показва прогнозата (домакин:гост), предсказания победител и изчислените точки.
- **Добавяне и редакция на прогнози** — форма, в която потребителят избира мач, въвежда резултат и потвърждава. Системата автоматично определя дали да се направи INSERT или UPDATE в базата. При въвеждане на негативни стойности за съществуваща прогноза — прогнозата се изтрива.
- **Автоматично изчисляване на точки** — при всяко зареждане на данните точките се преизчисляват за всяка прогноза спрямо реалния резултат. Класирането на потребителите се актуализира.
- **Страница "Всички мачове"** — преглед на всички мачове с реалните резултати, разгъваеми редове (expandable rows) с прогнозите на всеки участник и техните точки за конкретния мач.
- **Страница с Правила** — описание на правилата и таблица с коефициентите по фази.
- **Realtime обновяване** — Socket.io следи за промени в резултатите от External API; Supabase Realtime следи за промени в таблица `predictions`. При детектирана промяна данните се презареждат автоматично без действие от потребителя.
- **Групиране по фази** — мачовете са наредени по фаза (групова, 1/32, 1/16, четвъртфинали, полуфинали, финал). Фазите могат да се скриват/показват с един клик.
- **Групиране по кръгове** — груповата фаза е разделена на 3 кръга (`cicle_1`, `cicle_2`, `cicle_3`) с точни дати, за да се вижда кога са изиграни мачовете.
- **Тъмен режим** — превключване между светла и тъмна тема, запазено в `localStorage`.
- **Многоезичност (BG / EN)** — целият интерфейс е преведен. Датите и часовете се форматират спрямо избрания език (`bg-BG` или `en-US`).

### За администратори

- **Импорт на отбори** — администраторът качва JSON файл с отбори, системата ги записва в таблица `teams` в Supabase.
- **Импорт на мачове** — качване на JSON файл с мачове в таблица `matches`. Поддържа исторически данни от турнирите 2016, 2018, 2020, 2022 и 2024 (файловете са включени в проекта като `backup_20XX.json`).

### Допълнителни технически функции

- **Excel export** — изтегляне на таблицата с прогнозите като форматиран `.xlsx` файл. Файлът съдържа заглавен ред с имената и точките на участниците, подзаглавия (Д / Г / П / Т), обединени клетки за секциите и локализирани имена на колоните.
- **Пълен одит (Backup)** — всяка операция (добавяне, редакция, изтриване, пропуск) генерира уникален `event_id` (timestamp + random hex) и се записва в таблица `prediction_backup_events` с пълния payload — user_id, match_id, стойности на прогнозата, timestamp и евентуална грешка.
- **PWA поддръжка** — Angular Service Worker кешира assets и позволява приложението да работи офлайн или при слаба мрежа.

## Как работи приложението

### Потокът на прогнозата

1. Потребителят отваря страница `/add-prediction` и избира мач от падащо меню.
2. Въвежда домакин/гост резултат (полувреме и пълно време) и предсказания победител.
3. Системата (`AllPredictionsPredictionFlowService`) проверява:
   - Има ли вече прогноза за този мач от потребителя?
   - Стойностите валидни ли са (≥ 0)?
   - На база тези условия — прави INSERT (нова), UPDATE (съществуваща), DELETE (нулиране) или SKIP (пропуск).
4. Всяка операция се записва и в одит таблицата `prediction_backup_events`.

### Realtime синхронизация

Системата поддържа **два независими realtime канала** едновременно:

- **Socket.io** — свързва се към `https://simple-node-proxy.onrender.com` и слуша за `matchesUpdate` събитие. При промяна проверява дали данните са се изменили (hash сравнение) и само тогава задейства презареждане.
- **Supabase Realtime** — WebSocket абонамент за таблица `predictions`. При всяка INSERT / UPDATE / DELETE операция Angular компонентът автоматично презарежда прогнозите от базата.

### Изчисляване на точките

Точките се изчисляват в `AllPredictionsPointsService` при всяко зареждане — не се пазят предварително изчислени в базата. Логиката сравнява `home_ft` / `away_ft` / `winner` на прогнозата с реалния резултат на мача. Общите точки на потребителите се сумират и потребителите се сортират низходящо.

## Архитектура на кода

`AllPredictionsComponent` е главният оркестратор на приложението. За да се избегне монолитен компонент, логиката е разделена в 7 специализирани услуги:

```
AllPredictionsComponent  (orchestrator — зарежда данни, управлява state)
│
├── AllPredictionsPointsService
│     Изчислява точките за всяка прогноза спрямо реалния резултат.
│     Сумира общите точки по потребител и сортира класирането.
│
├── AllPredictionsMapperService
│     Трансформира сурови данни от Supabase/API в Bet редове за таблицата.
│     Форматира фази, кръгове, дати и часове по текущия език.
│     Определя към кой цикъл (cicle_1/2/3) спада даден мач по дата.
│
├── AllPredictionsRealtimeService
│     Отваря Socket.io връзка и Supabase Realtime абонамент.
│     Сравнява хешове на данните, за да избегне излишни презареждания.
│
├── AllPredictionsExportService
│     Генерира Excel файл (.xlsx) с всички прогнози.
│     Строи worksheet матрица с обединени клетки и локализирани заглавия.
│
├── AllPredictionsBackupService
│     Генерира уникален event_id за всяка операция.
│     Записва backup entry в Supabase и/или localStorage при грешка.
│
├── AllPredictionsPredictionFlowService
│     Решава дали да се прави INSERT / UPDATE / DELETE / SKIP.
│     Изгражда mutation payload и извиква SupabaseService.
│
└── AllPredictionsThemeService
      Чете theme-color и dark-mode от localStorage.
      Изчислява контрастен текстов цвят по YIQ формула.
```

### Споделени модели (`all-predictions.models.ts`)

| Тип | Полета | Описание |
|-----|--------|----------|
| `Match` | id, utcDate, stage, group, homeTeam, awayTeam, score | Мач от API или база |
| `Prediction` | id, home_ft, away_ft, home_pt, away_pt, winner, users, matches, teams, points | Прогноза с вложени релации |
| `User` | id, name_bg, name_en, total_points | Участник |

| `Bet` | row_index, match_day, match_time, group, stage, phase, home_team, away_team, score | Ред за показване в таблицата |
| `PredictionBackupEntry` | event_id, timestamp, action, user_id, match_id, payload, error_message | Одит запис |

## База данни

Схемата в Supabase се съдържа в папка `supabase-migrations/`. Всички таблици са защитени с **Row Level Security (RLS)** — само публичен (anon) достъп за четене, ограничен запис.

### Таблици

| Таблица | Ключови колони | Описание |
|---------|---------------|----------|
| `users` | id, name_bg, name_en | Участниците в играта |
| `teams` | id, name_bg, name_en | Футболни отбори (BG и EN имена) |
| `matches` | id, home_team_id, away_team_id, utc_date, stage, group, score_home, score_away, winner | Мачове с резултати |
| `predictions` | id, user_id, match_id, home_ft, away_ft, home_pt, away_pt, winner, utc_date | Прогнозите на участниците |
| `prediction_backup_events` | event_id, event_timestamp, action, user_id, match_id, prediction_id, column_index, input_value, payload, error_message | Пълен одит на промените |

### View: `predictions_with_users`

Денормализиран изглед, който обединява `predictions` с `users`, `matches` и `teams` в една заявка. Angular чете от него чрез Supabase `select` с вложени релации:

```sql
SELECT
    p.id, p.user_id, p.match_id, p.utc_date,
    p.home_ft, p.away_ft, p.home_pt, p.away_pt, p.winner,
    u.name_bg, u.name_en,
    ht.name_en AS home_team_en, ht.name_bg AS home_team_bg,
    at.name_en AS away_team_en, at.name_bg AS away_team_bg
FROM predictions p
    INNER JOIN users u ON p.user_id = u.id
    INNER JOIN matches m ON p.match_id = m.id
    INNER JOIN teams ht ON m.home_team_id = ht.id
    INNER JOIN teams at ON m.away_team_id = at.id;
```

> Подробно ръководство за view-то: [`SUPABASE_VIEW_GUIDE.md`](SUPABASE_VIEW_GUIDE.md)

## Правила и точкуване

### Подаване на прогнози

- Прогнозите се подават **най-късно 5 минути преди началото** на мача.
- При изпуснат краен срок **не може** да се подаде прогноза за дадения мач.
- Вече подадена прогноза може да се **редактира** до 5 минути преди мача.
- Прогнози могат да се подадат и по email: **ramsess90@gmail.com**

### Точкуване

| Резултат | Точки | Пример |
|----------|-------|--------|
| Точен резултат | **3** | Прогноза 2:1, реален резултат 2:1 |
| Правилна разлика + победител | **2** | Прогноза 2:1, реален резултат 3:2 |
| Само правилен победител / равенство | **1** | Прогноза 2:1, реален резултат 1:0 |
| Грешна прогноза | **0** | Прогноза 2:1, реален резултат 0:1 |

Точките се **умножават по коефициент** в зависимост от фазата:

### Коефициенти по фази

| Фаза | Коефициент | Финални точки за точен резултат |
|------|------------|---------------------------------|
| Групова фаза | ×1 | 3 т. |
| 1/32-финали | ×1.5 | 4.5 т. |
| 1/16-финали | ×1.5 | 4.5 т. |
| Четвъртфинали | ×2 | 6 т. |
| Полуфинали | ×2.5 | 7.5 т. |
| Мач за 3-то място | ×2.5 | 7.5 т. |
| Финал | ×3 | 9 т. |

## Теми и потребителски интерфейс

- **4 цветови теми** — Зелено, Червено, Синьо, Жълто. Изборът се пази в `localStorage` като hex стойност под ключ `theme-color`.
- **Тъмен режим** — включва се от header-а, записва се под ключ `dark-mode`. Приложен към фона и текстовете на таблицата.
- **YIQ контраст** — `AllPredictionsThemeService` автоматично изчислява дали текстът върху избрания цвят трябва да е черен или бял, по формулата: `YIQ = (R×299 + G×587 + B×114) / 1000`. Стойност ≥ 128 → черен текст; < 128 → бял текст.
- **Responsive** — при ширина < 768px (`IS_SMALL_SCREEN`) таблицата преминава в мобилен изглед.
- **Скриване на фази** — потребителят може да скрие/покаже фаза с един клик; изборът се пази в `localStorage` под ключ `hiddenGrops`.

## Администраторски достъп

Администраторският достъп се активира чрез URL параметър:

```
https://example.com/?set-admin=<admin-key>
```

Достъпът може да се деактивира чрез:

```
https://example.com/?remove-admin
```

`AdminService` проверява ключа спрямо `environment.adminKey`. При съвпадение записва хеш на ключа в `localStorage` под `admin_unlocked`. Всички последващи заявки към `adminGuard` проверяват само тази стойност — не се налага повторен вход.

За да се **деактивира** достъпът, `AdminService.lock()` изчиства записа от `localStorage`.

## Environment конфигурация

Проектът използва Angular environment файл `environments/environment.ts`. Трябва да го попълниш преди стартиране:

```ts
export const environment = {
  production: false,
  supabaseUrl: 'https://<your-project>.supabase.co',  // URL от Supabase проект → Settings → API
  supabaseKey: '<your-anon-key>',                     // anon/public ключ от Settings → API
  adminKey: '<your-secret-admin-key>'                 // произволна стойност, използвана за ?set-admin=
};
```

| Ключ | Откъде | Описание |
|------|--------|----------|
| `supabaseUrl` | Supabase → Settings → API → Project URL | Base URL на Supabase проекта |
| `supabaseKey` | Supabase → Settings → API → anon public | Публичен ключ — достъпът се ограничава от RLS |
| `adminKey` | По избор | Таен ключ за активиране на admin режим чрез URL |

> **Важно:** Никога не комитвай реален `adminKey` или Supabase service role ключ в хранилището. `supabaseKey` (anon) е публичен и сигурен за клиентски код, тъй като достъпът до данни се контролира от Row Level Security политиките в Supabase.
